v1 변경점
Why: 베이스라인 v0에서 발생한 다국어 혼입 및 역할 붕괴가 실제 시스템 적용을 불가능하게 만듦
What: 모든 모델에 강제 적용되는 공통 출력 규칙(Common Rules) 추가(한국어 전용, 질문 1개 등)
v2 변경점
Why: v1에서 안정성은 확보했으나 스타일별 차이가 없고 질문이 STAR 구조로 인해 길어짐
What: 압박형(검증/수치 확인)과 친화형(경험 구체화/유도) 스타일 규칙 분리 정의
v3 변경점
Why: 규칙의 복잡도에 따른 모델의 통제력을 확인하고 어떤 규칙이 언어 붕괴를 유발하는지 파악 필요
What: 최소 규칙, 실패 기반 규칙, 스타일 전용 규칙으로 구분하여 실험적 비교 수행
v4 변경점
Why: 이전 버전들에서 첫 질문이 신입 지원자가 답변하기 어려운 경력자 수준으로 생성됨
What: 면접 상태가 첫 질문(FIRST_QUESTION)일 때 신입 기준의 탐색형 질문을 던지도록 별도 지시문 추가
v5 변경점
Why: 프롬프트 지시사항이 누적되면서 모델이 스타일 규칙보다 면접의 흐름을 놓치는 현상 발생
What: 질문 스타일보다 면접 상태(State)를 최상단에서 제어하도록 구조 변경, 단일 키워드 기반 질문 강조
v6 변경점
Why: 프롬프트 수정만으로는 질문 품질 개선이 정체되어 규칙을 얼마나 잘 지키는지 측정이 필요해짐
What: 스타일 가이드라인을 정량적으로 검증 가능한 형태로 명문화하고 QA 자동 평가 체계 구축
v7 변경점
Why: 규칙 설명만으로는 모델이 의도한 문체와 스타일을 충분히 재현하지 못함
What: 압박형과 친화형 각각에 실제 면접 질문 예시(Few-shot)를 포함하여 모델이 패턴을 학습하게 함
v8 변경점
Why: 친화형의 장문 폭주와 소형 모델의 중국어 출력을 시스템적으로 차단할 필요가 있음
What: 규칙 위반 시 재생성(Re-generation)을 위한 로직 설계 및 상태별 예시 문장 보강
v9 변경점
Why: 잦은 리트라이로 인한 응답 지연(Latency) 문제와 VRAM 관리 효율성 확보가 필요함
What: 검증 기준을 필수 오류 차단 위주로 축소하고, 모델 순차 로딩 및 메모리 회수 파이프라인 안정화
v10 변경점
Why: 스타일이 안정화되자 문장 패턴이 고착화되고 특정 표현이 반복되는 현상 발견
What: 페르소나 정의를 문장 스타일 지시에서 '행동 규칙(인정 1회, 모호성 탐지 1회 등)' 중심으로 재설계
v11 변경점
Why: 리트라이 로직이 생성 분포와 맞지 않아 불필요한 실패가 누적되는 문제 해결 필요
What: 프롬프트와 검증기 간의 정합성을 조정하고 상태 기반 프롬프트의 강제력 강화
v12 변경점
Why: 프롬프트 튜닝만으로는 해결할 수 없는 모델별 본질적 차이(7B 언어 혼입, 9B 장문화) 인정
What: 프롬프트 수정을 멈추고 고정 테스트셋 기반의 실패 분석 중심 시스템으로 전환, 검증기를 제어가 아닌 분석 도구로 재정의
